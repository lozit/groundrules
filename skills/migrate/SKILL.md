# /groundrules:migrate

You will update a groundrules-bootstrapped project (including projects bootstrapped by the pre-1.0 plugin, named starter-kit) to the current plugin version.

If `$ARGUMENTS` contains `--dry-run` (or `dry-run`), run all analysis phases but **write no file**: end with a report only.

## Phase 0 ‚Äî Plugin update check (best-effort, never blocking)

1. Read `version` from `${CLAUDE_PLUGIN_ROOT}/.claude-plugin/plugin.json` ‚Üí INSTALLED.
2. Run via Bash with a **short timeout (~3s)**: `git ls-remote --tags --refs --sort=-v:refname https://github.com/lozit/groundrules.git 'v*' | head -1`
3. Extract the tag (`refs/tags/vX.Y.Z`). If semver-greater than INSTALLED, warn (informational, don't stop):
   > üì¶ groundrules vX.Y.Z is available but vINSTALLED is installed ‚Äî migrating now will only bring the project up to vINSTALLED. Recommended: update the plugin first ‚Äî `/plugin marketplace update claude-code-groundrules` (catalog only), then **reinstall** (`/plugin install groundrules@claude-code-groundrules`) and **restart Claude Code** ‚Äî then re-run `/groundrules:migrate`. (Marketplace update alone doesn't update the installed plugin.) Continuing with vINSTALLED is fine too.
4. **Fail silent**: on timeout, no network, or any error, continue without mentioning the check. This is the only network access in this skill and it is best-effort (cf. ADR 0015).

## Phase 1 ‚Äî Version detection

1. `.groundrules.json` must exist in the cwd. **Legacy**: if only a pre-1.0 `.starter-kit.json` exists, read that instead (version key `starterKitVersion`) ‚Äî the V1.0 rename below will move it. If neither exists: *"This project was not bootstrapped with groundrules, nothing to migrate."* Stop.
2. Read the state file ‚Üí extract:
   - `groundrulesVersion` (legacy file: `starterKitVersion`) ‚Üí **OLD**
   - `answers` ‚Üí original interview answers
   - `generatedFiles` ‚Üí list of files generated at bootstrap
3. Read `${CLAUDE_PLUGIN_ROOT}/.claude-plugin/plugin.json` ‚Üí extract `version` ‚Üí **NEW**.
4. Compare:
   - **OLD == NEW** ‚Üí *"Project up to date at version NEW. Nothing to migrate."* Stop.
   - **OLD > NEW** (semver) ‚Üí *"Project version (OLD) is newer than the installed plugin (NEW). Refusing to downgrade."* Stop.
   - **OLD < NEW** ‚Üí continue.

## Phase 2 ‚Äî Show what changes

Read `${CLAUDE_PLUGIN_ROOT}/CHANGELOG.md`. Extract and show the entries between OLD and NEW so the user understands what they'll get.

> **Migrations that rename/move files**: some versions rename or relocate generated files ‚Äî e.g. V0.7 renamed `docs/00-VISION.md` ‚Üí `docs/VISION.md` and `brief/00-INTENT.md` ‚Üí `brief/INTENT.md`; V0.9 moved `media/` ‚Üí `docs/media/`; V0.11 renamed the `brief/` folder ‚Üí `intake/` (ADR 0014). When migrating across such a version, detect the old path on disk and offer to `git mv` it to the new path (never duplicate; never delete without confirmation). Renames **chain**: a pre-V0.7 `brief/00-INTENT.md` migrating past V0.11 lands directly at `intake/INTENT.md` (one `git mv` to the final path). For `media/` ‚Üí `docs/media/`: if a top-level `media/` exists and was the starter-kit one, offer the move; if the project has its own unrelated `media/`/`public/`, leave it and just create `docs/media/`. For `brief/` ‚Üí `intake/`: offer `git mv brief intake` (whole folder), then fix the paths in `generatedFiles` and any `intent.source`-style references in `.groundrules.json`; also flag stale `brief/` references in the project's own `CLAUDE.md`/`README.md`/docs for the user to update (or offer to update them if they carry the starter-kit signature).

### V1.0 ‚Äî the plugin was renamed starter-kit ‚Üí groundrules (ADR 0017)

When OLD < 1.0.0, the project was generated by the plugin under its former name **starter-kit**. After the regular per-file arbitration (phases 3-6), apply the **rename pass** (each step after confirmation, `git mv` for tracked files):

1. **State file**: `git mv .starter-kit.json .groundrules.json`, then rename the key `starterKitVersion` ‚Üí `groundrulesVersion` inside it (all other keys unchanged).
2. **Signatures**: in every file of `generatedFiles` that carries `<!-- generated-by: starter-kit vX.Y.Z -->`, rewrite the signature line to `<!-- generated-by: groundrules vNEW -->`. One grouped confirmation for all files ("rewrite N legacy signatures?"), not one question per file. Files the user chose to keep untouched in phase 5 still get the signature rewrite offer ‚Äî it's a one-line change, not a content overwrite.
3. **Command prefix**: inform the user that all slash commands changed: `/starter-kit:<skill>` ‚Üí `/groundrules:<skill>`. If the project's own `CLAUDE.md`/docs mention `/starter-kit:` commands (grep them), offer to update those mentions.
4. **Stale name references**: flag remaining "starter-kit" mentions in the project's generated docs (`README.md` structure section, notes "bootstrapped with starter-kit"...) and offer to update them to groundrules + the new repo URL (`https://github.com/lozit/groundrules`).
5. **Plugin reinstall reminder** (recap): the marketplace is now `claude-code-groundrules` and the plugin `groundrules` ‚Äî the user should `/plugin marketplace add https://github.com/lozit/groundrules` + `/plugin install groundrules`, and may remove the old `starter-kit` plugin/marketplace entries.

## Phase 3 ‚Äî Analyze tracked files

For each file listed in `generatedFiles`:

1. Identify the template that produced it. Use the same mapping as `skills/bootstrap/SKILL.md` Phase 5. Templates are English-only with a single `.tpl` per file (no language variants).
2. Read `${CLAUDE_PLUGIN_ROOT}/skills/bootstrap/templates/<template>` (current version).
3. Substitute the placeholders with `answers` (same logic as bootstrap Phase 5).
4. Read the file on disk.
5. Compare via Bash `diff -u <(printf '%s' "$current") <(printf '%s' "$generated")`:
   - **Identical** ‚Üí note "already up to date"
   - **Differs** ‚Üí note "to arbitrate" + keep the diff handy

> **Legacy projects (pre-V0.8, bilingual)**: if `answers.lang` is present (`fr`/`mix`), the project was generated by an older bilingual plugin. The current plugin is English-only ‚Äî there is no FR template to diff against. Do NOT overwrite French content with English; report these files as "language change ‚Äî manual review" and, if the user wants to switch to English, offer to write the new English template as `<file>.new` for manual merge.

## Phase 4 ‚Äî Detect newly available files

For each template in the current plugin that would produce a file given `answers` (respecting the `HAS_*` flags) **but that is NOT in `generatedFiles`**:

- It's a new template introduced in a recent version ‚Üí offer to create it.

## Phase 5 ‚Äî Recap and arbitration

Show a recap table:

```
| File                   | State             | Proposed action      |
|------------------------|-------------------|----------------------|
| CLAUDE.md              | content modified  | show diff then choose |
| docs/LEARNINGS.md      | identical         | skip                 |
| docs/ARCHITECTURE.md   | identical         | skip                 |
| (new) docs/...         | template added    | create? (optional)   |
```

For each "content modified" file, ask via `AskUserQuestion` (group by 3-4):

- `See the diff` ‚Äî show the diff via Bash `diff -u`, then re-ask
- `Overwrite with the new template` ‚Äî destructive, but updates
- `Keep my file` ‚Äî skip
- `Save the new one as .new` ‚Äî write the new content to `<file>.new` for manual merge by the user

For each "new template available": `Create` / `Skip`.

## Phase 6 ‚Äî Apply decisions

> If `--dry-run`: skip this phase, go straight to phase 8 with a "would have done:" report.

For each "Overwrite" ‚Üí `Write` the new content.
For each "Save as .new" ‚Üí `Write` to `<file>.new`.
For each "Create" ‚Üí `Write` the new file.
For each "rename" (legacy file name) ‚Üí `git mv` to the new name (after confirmation).

## Phase 7 ‚Äî Update `.groundrules.json`

Update:
- `groundrulesVersion` ‚Üí NEW
- `generatedFiles` ‚Üí updated list (add newly created files, fix renamed paths)
- If `answers.lang` exists and the user migrated content to English, drop the obsolete `lang` key (the plugin no longer uses it).
- Append (or create) a `migrations` array entry:
  ```json
  {"from": "OLD", "to": "NEW", "at": "YYYY-MM-DD"}
  ```

## Phase 8 ‚Äî Final recap

Show:
- ‚úÖ Files updated (overwrite)
- ‚úÖ Files created (new templates)
- üîÅ Files renamed (legacy ‚Üí new name)
- üìÅ `.new` files to merge manually
- ‚è≠Ô∏è Files left as-is (user choice)
- üìã Next steps:
  1. Review the `.new` files and merge in what you like
  2. Flesh out the project-specific sections (Setup/Build/Test commands, stack, etc.) in CLAUDE.md if they were regenerated
  3. Commit when ready ‚Äî if `.groundrules.json` has `policies.noAiAttribution = true` (or a project/global CLAUDE.md forbids AI attribution), the suggested commit message must contain **no** AI attribution marker (`Co-Authored-By`, "Generated with Claude Code"‚Ä¶), even if a default agent guideline would add it.

**NEVER commit automatically.**

## Important rules

- **No overwrite without an explicit user choice** (phase 5).
- **Refuse downgrade**: if the project version is newer than the installed plugin, stop.
- If a template referenced in `generatedFiles` **no longer exists** in the current plugin (file removed in a recent version), don't touch the file; report it in the recap.
- If `--dry-run`: no writes, just a report ‚Äî useful to understand before committing.
- The diff must be readable: prefer `diff -u` (unified) over raw `diff`.
