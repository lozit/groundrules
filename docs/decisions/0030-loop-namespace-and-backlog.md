# 0030 — `loop/` as the loop's namespace; a dedicated backlog referenced by `PLAN.md`

**Date**: 2026-06-14
**Status**: Accepted

## Context

[M1 brick 2](../prd/loop-scaffolding-bootstrap-adopt.md) productizes the validated maker/verifier loop
contract (brick 1, `docs/prototypes/loop/`) as opt-in scaffolding generated by `bootstrap`/`adopt`. Two
structural decisions had to be settled before building, and they turned out to be one:

1. **Where do the generated loop artifacts live** in a user project — a root `loop/`, a hidden
   `.groundrules/loop/`, or `docs/loop/`?
2. **What backlog does the generated loop read** before `/groundrules:realize` (brick 3) exists to
   produce a partitioned `[loop]`/`[supervised]` backlog — a `[loop]`-tagged section of `PLAN.md`, or a
   separate file?

The artifacts are not durable human documentation (ADR/LEARNINGS/VISION) — they are **executable
tooling + agent-facing prompts + volatile loop state** (a backlog, parked blocks, accumulated lessons).
An autonomous loop reads them, mutates some of them, and commits — so the boundary between *what the
loop owns* and *what the human curates* (`PLAN.md`) is the real question.

## Decision

**`loop/` is a single, visible, top-level directory that is the loop's complete working namespace.** It
holds the prompts (`maker.md`, `verifier.md`, `LOOP.md`), the capped runner (`run-loop.sh`), the loop's
**own backlog (`loop/backlog.md`)**, its parked decisions (`loop/blocked.md`), optional accumulated
lessons (`loop/lessons.md`), and a `loop/.gitignore`.

**The loop reads its own `loop/backlog.md` — never `PLAN.md` directly.** `loop/backlog.md` contains only
**loop-safe tasks**; it is hand-filled for now and will be *populated by `/groundrules:realize`* later
(brick 3). `PLAN.md` carries a one-line **pointer** to it (*"Loop backlog → `loop/backlog.md`
(loop-safe tasks only)"*) and **does not duplicate** its tasks.

Ownership boundary: **the loop owns `loop/`; the human owns `PLAN.md`.**

## Alternatives considered

- **`.groundrules/loop/` (hidden dir)** — rejected: (a) it introduces a confusing sibling to the
  existing `.groundrules.json` *file* at the root; (b) hiding the maker/verifier prompts in a dotdir
  discourages the user from owning and tuning them, contradicting *handoff-not-gospel* — groundrules
  sells comprehension, not a black box.
- **`docs/loop/`** — rejected: wrong altitude. `docs/` is durable, human-readable project memory; a
  `run-loop.sh` executable does not belong there.
- **A `[loop]`-tagged section inside `PLAN.md`** — rejected on safety and principle: (a) an autonomous
  loop would have to *filter* "only `[loop]`-tagged" perfectly; a miss grabs a `[supervised]`/decision
  task — the exact over-reach the loop guards against. A dedicated file makes the "loop-safe only"
  invariant **structural** (which file), not a tag-filter the agent must honor. (b) The loop checks off
  and commits autonomously; mutating `PLAN.md` races with the human's curation surface. (c) Our own
  "compare by altitude" learning (`docs/LEARNINGS.md`) says a *volatile, agent-touched* artifact must
  not be merged by name with a *durable, human-curated* one — separate by lifecycle, reference don't
  duplicate (the same resolution we used for superpowers' `PLAN.md` friction).

## Consequences

### Positive
- A clean ownership boundary the loop-safety story rests on: the loop's blast radius is *structurally*
  confined to `loop/`, not policed by a convention.
- Consistent with the existing top-level **`intake/`** (a working dir that isn't `docs/`).
- Gives `/groundrules:realize` (brick 3) a clear write target (`loop/backlog.md`) — a forward
  constraint, recorded now, not a decision deferred.
- Defuses the "inert scaffolding without `realize`" risk: `loop/backlog.md` ships with hand-fill
  instructions and is later auto-filled.

### Negative / Tradeoffs
- An extra top-level `loop/` directory for projects that opt in (off by default; the user owns the repo
  and the paths live in `LOOP.md`, easy to relocate).
- Two backlogs (`PLAN.md` + `loop/backlog.md`) carry a confusion risk — mitigated by the one-line
  pointer and the "reference, don't duplicate" rule.
- `loop/` is a slightly generic name; `bootstrap`/`adopt` must detect a pre-existing `loop/` and apply
  the usual skip / `.new` resume-safety.

### Neutral
- `blocked.md` lives at `loop/blocked.md` (cohesion — all loop state in one place). Block visibility is
  carried by the runner's stop message + a `CLAUDE.md` convention line ("on a block, triage
  `loop/blocked.md`"), not by root placement.
- Single loop / single backlog for now; per-feature loops (multiple backlogs under `loop/`) are not
  precluded but not built.
