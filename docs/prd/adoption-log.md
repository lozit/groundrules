# PRD — adopt → ADOPTION-LOG.md (field → plugin feedback channel)

> Product Requirements Document for a single feature. Written **before** building, so the agent
> builds the right thing — not a coherent surprise. Validate it (and answer the open questions)

**Date**: 2026-06-14 · **Status**: validated

## Problem

When the maintainer runs groundrules (`adopt`) on a real project and it **doesn't do quite what they
wanted**, that signal is lost: the adopt recap is ephemeral, and `.groundrules.json` is machine JSON,
not a shareable account of *what was there* and *what groundrules did*. There is no artifact the
maintainer can **annotate and share back** to improve the plugin.

The need: a human-readable **log of the adopt run** that gives enough context (before-state + actions
taken) that the maintainer's **remarks** ("here groundrules should have done X") are actionable when
harvested back into the groundrules repo. A **field → plugin feedback channel.**

## Success criteria

- After an `adopt` run, the user can opt in to generate `docs/ADOPTION-LOG.md` that records, in plain
  language: (1) **what was detected** (before), (2) **what groundrules did** (and *why*), (3) a
  **Remarks** section for the user to fill.
- The log is **self-contained**: a reader (future me / the groundrules maintainer) understands the
  adoption from the file alone, without re-scanning the project.
- It is **frozen + dated** and **resume-safe**: re-running `adopt` never silently overwrites it
  (skip / overwrite / `.new`).
- It is clearly **not** an AGENT-EVALS (agent behaviour) nor an apply-best-practices (external recs)
  — it is about *what the adopt run did*.

## Scope

**In scope**
- An **opt-in step at the end of `adopt`** (one `AskUserQuestion`) that offers to write the log.
- A new template `ADOPTION-LOG.md.tpl` (signature + the three sections).
- The skill populates it from data it **already has**: the scan results, the user's `answers`, and
  `.groundrules.json` `generatedFiles` / `adoptedFiles` / `skippedFiles` / `migratedFiles` /
  `adoptionMode`. Key decisions get a one-line *why* (e.g. content-aware omissions, consolidate vs
  map-in-place, deferral to a managed CLAUDE.md).
- A **Remarks** section pre-seeded with prompts ("Anything groundrules got wrong or you'd change?
  Note it inline, then share this file back to improve groundrules").
- `docs/ADOPTION-LOG.md`, frozen + dated; resume-safe.
- README "What's generated" conditional row; CHANGELOG + PLAN.

**Out of scope** (explicit)
- **Not generated by `bootstrap`** (greenfield has no "what was there" — the recap covers it).
- **No structural audit / improvement recommendations** about the *project* (that was the earlier
  misread; and external recs are `apply-best-practices`' job).
- **Not a new skill** — it's an `adopt` output. (A standalone re-audit is a possible later idea.)
- No network; no machine-readable export beyond the existing `.groundrules.json`.

## Constraints

- **Template over code** (ADR 0002): `ADOPTION-LOG.md.tpl` is plain `{{KEY}}` text; the skill fills it.
- **English-only** (ADR 0012); carries the `generated-by` signature.
- **Never overwrite** without confirmation (resume mode — ADR's idempotence rule).
- **Honest content** (Posture): record what was skipped/deferred and *why*, not a success-only story.

## Build plan

Ordered steps, each with a validation point.

1. **Design `ADOPTION-LOG.md.tpl`** — sections: `## What was here` (`{{DETECTED}}`), `## What
   groundrules did` (`{{ACTIONS}}` — generated/mapped/skipped + key decisions with *why*),
   `## Remarks (fill in, then share back)` (static prompts). Placeholders: `{{PROJECT_NAME}}`,
   `{{DATE}}`, `{{DETECTED}}`, `{{ACTIONS}}`. Validate: the three sections map to data `adopt` has.
2. **Add the opt-in Call to `adopt/SKILL.md`** (after generation): offer the log; build `{{DETECTED}}`
   from the scan, `{{ACTIONS}}` from the run + `.groundrules.json`; write `docs/ADOPTION-LOG.md`
   (resume-safe). Validate: dry-read; the log is self-contained and honest about skips/deferrals.
3. **README** — conditional row in "What's generated" (`docs/ADOPTION-LOG.md` — when adopting, opt-in).
4. **CHANGELOG `[Unreleased]`** + PLAN move (Up next → done).
5. **E2E** — fresh-subagent: on a fixture project (some existing structure), follow the new Call with
   canned scan/answers → produce `ADOPTION-LOG.md`; check it captures before + actions + a Remarks
   section, is self-contained, and never overwrites an existing log.

## Risks

What could go wrong, and the mitigation.

- **Overlap confusion with AGENT-EVALS / apply-best-practices.** Mitigation: the log's own preamble
  states what it is (a record of *this adopt run* for feedback) and points to the other two.
- **Success-only / vacuous log.** Mitigation: the template *requires* the skip/defer reasons and the
  Remarks prompts; honest by construction (Posture).
- **Staleness** (the log ages as the project evolves). Mitigation: it's a **dated, frozen snapshot of
  the adopt run** by design — not a living doc; that's stated in its header.
