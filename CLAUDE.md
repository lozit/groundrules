# CLAUDE.md — groundrules (meta)

**Important**: this file guides Claude when it **develops the groundrules plugin** (formerly starter-kit, renamed at V1.0.0 — ADR 0017). It is **not copied** into user projects — `skills/bootstrap/templates/CLAUDE.md.tpl` (content-aware tailored against a global, cf. ADR 0029) serves that purpose.

## Nature of the repo: two disjoint layers

This repo is both:

**Layer A — Plugin sources** (what is published):
- `.claude-plugin/plugin.json` — manifest
- `.claude-plugin/marketplace.json` — mono-plugin marketplace for local install
- `skills/<name>/SKILL.md` — logic of each `/groundrules:<name>` slash command
- `skills/bootstrap/templates/*.tpl` — Markdown templates used at generation time

**Layer B — Project docs of the plugin itself** (self-application / dogfood):
- `README.md` — public install and usage doc
- `CLAUDE.md` — this file
- `docs/`, `intake/`, `docs/media/`, `PLAN.md` — generated by the plugin applied to itself

The two layers **never overlap** at the path level. When you change something, ask yourself: "am I touching the plugin sources (A) or the project docs (B)?".

## Testing the plugin locally

```bash
# Install the local marketplace
claude
# then in Claude Code:
/plugin marketplace add /Users/guillaumeferrari/Projets/groundrules
/plugin install groundrules@claude-code-groundrules
```

Alternative for fast iteration:
```bash
claude --plugin-dir /Users/guillaumeferrari/Projets/groundrules
```

Test from-scratch:
```bash
mkdir /tmp/test-bootstrap && cd /tmp/test-bootstrap
claude
# then: /groundrules:bootstrap
```

Reloading after a change: a **new skill directory** requires a full restart of Claude Code; editing an existing `SKILL.md`/template is picked up by `/reload-plugins` (or `--plugin-dir` on the next launch).

## Template conventions

- Every template carries `<!-- generated-by: groundrules v1.4.0 -->` at the top.
- Placeholders in `{{KEY}}` format (plain text substitution, no template engine).
- Available placeholders defined in `skills/bootstrap/SKILL.md` (Phase 5).
- **English-only**: one `.tpl` per file, no language suffix (cf. ADR 0012). `gitignore.minimal` and `adr-template.md` are plain files.

## Git workflow

Conventions recorded in [ADR 0028](docs/decisions/0028-git-workflow-conventions.md).

- **Branching — trunk-based on `main`** (solo maintainer): changes land directly on `main`, tags `vX.Y.Z` mark releases, no feature branches. Deliberate, not an oversight: the generated template tells users to *state their own model* (it no longer hard-codes "feature branch for non-trivial changes"); ours is trunk-based.
- **Commits — at natural boundaries** (a completed chunk), Conventional Commits, **not** one mega-commit per release nor one per trivial change. Tag + finalize `CHANGELOG` at release. The commit message **references** the relevant `CHANGELOG` section rather than re-listing it. **Commit/push only on explicit request.**
- **AI attribution — attribute by default, defer to any forbidding rule.** Add `Co-Authored-By: Claude …` on commits, **unless** a global/enterprise `CLAUDE.md` instruction or `.groundrules.json` `policies.noAiAttribution: true` forbids it (same logic as ADR 0011). For this repo: **attribute** (no rule forbids). Never rewrite history to "fix" past attribution.

## Versioning

- **Batch small changes** — do NOT cut a version per change. Apply the change and add a line under `CHANGELOG.md` `## [Unreleased]`. Bump version, sweep signatures, tag, and create a GitHub release **only when explicitly asked** ("release" / "ship vX.Y.Z").
- At release time: bump `version` in `.claude-plugin/plugin.json` AND `.claude-plugin/marketplace.json`, bump the `<!-- generated-by: groundrules vX.Y.Z -->` signature in all templates/docs consistently, finalize the `## [Unreleased]` heading to `## [X.Y.Z]`, add a `.groundrules.json` migration entry, then commit + push + `git tag -a vX.Y.Z` + `gh release create`.
- **Also at release time, review the `README.md`** on three fronts; a release that changes behavior but not these is incomplete:
  - **Skill-list drift** (mechanical): diff `ls skills/` against the README `## The workflow` list — every skill dir must appear there. Also check that any new generated output (a new `*.tpl` → a file/folder) is reflected in `## What's generated`.
  - **Justification drift**: its two sections `## What the research says` and `## Established practices we adopt` — did this release add/change a choice (a skill, a posture, a convention)? If so, add/update the matching row and its `## References` entry.
  - The product-side equivalent of the skill-list check is the `checkpoint` skill's 4th bucket ("shipped a user-facing surface → sync the doc"); this meta rule is the dogfood mechanical version for *our* README.
- Keep `CHANGELOG.md` up to date (Keep-a-Changelog) — `[Unreleased]` is the accumulator between releases.

## When to document

### ADR — `docs/decisions/`
Any structural decision about the plugin (template format, new placeholder, new skill phase...) → an ADR.

### LEARNINGS — `docs/LEARNINGS.md`
Claude Code plugin format gotchas, surprising `AskUserQuestion` behavior, `gh`/`glab` gotchas...

### PLAN.md
Ongoing tasks on the development of the plugin itself.

### Capture at checkpoints
At a work boundary the agent can see — **before a push/tag/release** (most reliable) or a completed milestone — propose capturing proactively: **decided** → an ADR, **learned / blocked 30+ min** → `docs/LEARNINGS.md`, **agent mistake/hallucination/drift** → `docs/AGENT-EVALS.md` + a guard (cf. ADR 0022). The user can also run `/groundrules:checkpoint`. Capture in the repo, not in memory.

## Plugin workflow (mental model)

- **Template over code**: plain text `{{KEY}}` substitution in templates, never application logic nor a template engine (cf. ADR 0002).
- **Dogfooding**: the plugin applies to itself. `groundrules/` (this repo) contains its own layer B (docs/, intake/, PLAN.md...) generated by `/groundrules:bootstrap`.
- **Offline-first**: only `/groundrules:apply-best-practices` requires the internet (WebFetch). The rest works offline — `bootstrap`/`adopt`/`migrate` only attempt a best-effort update check (3s timeout, silent failure, cf. ADR 0015).
- **User handoff**: a generated `CLAUDE.md` is a **starting point**, not absolute truth. The user edits and enriches it per their project needs.

## Posture (dogfood of ADR 0026)

- **Push back**: challenge a plan/template change that's off-strategy, technically wrong, or inconsistent with an ADR/LEARNINGS. Surface tradeoffs; ask before guessing; don't be sycophantic.
- **Stay reversible**: confirm before hard-to-undo actions; commit only on explicit request; `verify-bootstrap` is post-hoc, not a runtime guard (ADR 0025).

## Don't

- Don't put logic in `plugin.json` (just metadata)
- Don't make a single giant `AskUserQuestion` in a SKILL → group by theme (3-4 questions max per call)
- Don't generate an opinionated `.gitignore` from an unconfirmed stack → minimal by default
- Don't overwrite a user file without explicit confirmation (resume mode)
- Don't forget that `disable-model-invocation: true` hides the skill from auto-invocation: only the slash command works

## Tech stack

None. Pure Markdown + JSON. The "code" lives in the SKILL.md instructions that Claude executes on demand.
